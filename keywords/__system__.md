The following prompt uses a structured, HTML-like tag language to specify implementation requirements. Each tag is a typed instruction block with a name, optional id, optional reference, and a body. Treat every tag as a binding directive: read its body carefully, follow its rules exactly, and produce output that fully satisfies its constraints. Tags may be nested — inner tags inherit the scope of their parent. Where a tag says "do not", "must not", or "strictly forbidden", treat the constraint as absolute. The tag glossary below defines the authoritative meaning of each tag you will encounter.

---

- **file_context** tag: scopes everything nested inside to the single source file identified by the path attribute. Apply every nested directive when implementing that file — its body, data structures, functions, and constraints together form the complete specification for it. Do not apply file-local constraints to other files unless they are restated there.
- **folder_context** tag: scopes everything nested inside to the folder identified by the path attribute. The context and constraints declared here apply to the folder's entire subtree — every nested file_context and folder_context inherits them. Apply them to all code generated under this folder.
- **reusable_automation_script** tag: action is registered as a macro behavior. Whenever the described condition is met or this action is referenced by name in other blocks, execute the following steps exactly as described in the body.
- **application_context** tag: application related context. Use the details inside to understand the domain, purpose, and overall structure of the application being built. Let this context inform all naming, architecture, and behavioral decisions across the implementation.
- **argument** tag: function or procedure argument that describes input. Honor the declared name, type, and any described constraints or default values exactly — do not rename, reorder, or alter the type.
- **prohibited_anti_patterns** tag: describes anti-patterns and "DO NOT" and "MUST NOT" conditions strictly to follow. Treat every listed item as an unconditional prohibition — do not apply the pattern anywhere in the generated output, even if it appears in provided examples or seems locally convenient.
- **user_interface_block** tag: a named, reusable UI component or layout section (e.g. card, panel, widget, sidebar). Implement the component exactly as described in the body — match the declared structure, child elements, and behavior. Do not add undeclared controls or omit declared ones.
- **compilation_and_testing_pipeline** tag: all generated code, configuration files, and project structure must remain compatible with these pipelines. Do not generate anything that breaks them.
- **approved_dependency_whitelist** tag: You are strictly forbidden from installing or importing any package outside of this list for the current scope. If you need extra dependency - ask first.
- **table_column** tag: defines a single column within a table — its name, data type, display label, and any constraints (nullable, unique, default value). Implement the column exactly as declared. Do not rename it, change its type, or add undeclared constraints.
- **data_structure** tag: implement the data model with this exact schema. Do not alter field names, change types, add undeclared fields, or omit any field listed here. This is the authoritative blueprint for this structure throughout the codebase.
- **error** tag: describes how to throw exact error types under the described conditions only. Do not substitute, wrap, or rename them. For every error listed inside, emit at least one regression test that fails without the guard and passes with it.
- **few_shot_example** tag: demonstrates the required implementation pattern. Replicate this structure, naming conventions, and style exactly as described.
- **field** tag: declares a single named property within a data structure or form — including its type, optionality, and any validation rules. Implement it exactly as specified. Do not rename it, change its type, or add undeclared constraints.
- **logic_flow** tag: implement the following logical sequence step-by-step, do not skip any code validations.
- **user_interface_form** tag: a form surface that collects structured user input. Implement every declared field, label, validation rule, and submission behavior exactly as described. Do not add undeclared fields or omit declared ones.
- **function_contract** tag: implement the function according to the binding contract below. Every argument, the return type, each error condition, and every step of the logic flow are mandatory — do not skip, reorder, rename, or approximate any of them.
- **enforced_patterns** tag: apply every pattern and practice listed inside in all generated code without exception. These are validated, required standards for this codebase — do not substitute alternatives, even equivalent-seeming ones.
- **user_interface_image** tag: an image element to render in the UI. Use the declared source, alt text, dimensions, and display properties exactly as specified. Do not substitute a different asset or omit accessibility attributes.
- **environment_runtime_and_language** tag: write all code strictly targeting the language and runtime specification. Apply the correct module syntax, standard library APIs, and runtime-specific idioms throughout. Do not use syntax, features, or tools that do not belong to this target.
- **reusable_library_context** tag: context for a shared library package (internal or external). Before writing any code that uses this library, understand its declared API surface, versioning constraints, and usage patterns described in the body. Do not reimplement what the library already provides.
- **reusable_module_context** tag: context for a reusable module within the codebase. Read the body to understand its responsibilities, exports, and expected consumers. Reuse its existing functionality — do not duplicate logic it already owns.
- **reusable_namespace_context** tag: All code in this scope belongs to the current namespace — emit it under the target language's namespacing construct (package, namespace, or module path) with this as the qualified name and import root.
- **read_it** tag: before writing any code that touches this reference, open and read the file(s) at the path above and analyze them carefully. Mirror their existing formatting, export patterns, and error-handling architecture. Do not guess at their contents, and do not reimplement what they already provide. If you already read it just in the current execution context then remember what you read and consider it.
- **static_asset** tag: read this data asset definition to understand its structure and access patterns.
- **return** tag: function or procedure returned result that describes output. Honor the declared type and shape exactly — do not alter the structure, add extra fields, or return a different type under any code path.
- **table_row** tag: defines the shape and content of a single row within a table. Implement the row's fields and their values exactly as declared. Do not add undeclared fields or omit any listed ones.

- **table** tag: defines a tabular data structure or UI table component — including its columns, row shape, and display or persistence behavior. Implement it exactly as declared. Do not alter column order, rename columns, or add undeclared behavior.

Implement only the elements declared below — do not add components, fields, columns, controls, or decorative elements that are not listed, and do not omit any that are.
